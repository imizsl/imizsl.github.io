管理上的思考

发表于 2018-05-24 | 分类于 管理 |

几年前，我在网上看到一篇文章，讲诉的一个企业的领导带领大家共同奋斗，共同成长的心历路程，当时很感动，觉得以后自己做管理也需要有老大的风范。

我的失败案例

我在家里排行老二，我有个姐姐，一个弟弟。因为家里不富裕，姐姐很早就开始工作了。我的学习成绩不错，弟弟的学习成绩不好，周末的时候，我就经常以“老大”的角色辅导弟弟的功课。为了让弟弟学多掌握些计算方法，我经常给弟弟出一些自以为用到各种技巧的难题（现在想来是偏题）让他做，可是弟弟的成绩也没有什么提升。现在想来，我忽略了循序渐进的引导，沉浸在自我炫技的满足中，当初可能还是我害了他。

大学毕业后参加工作，我很用功，半年内学会了大多数技术文档，很快的在工作里脱颖而出做了组长，做了组长后带了两个妹子，她们有疑难杂症就来找我，我总是能帮她们解决问题，自我感觉良好，很有成就感，问题总是不断，我却也从来没有考虑如果让她们在解决问题中成长，虽然我们当时同事的情感很好，但是现在想来，当初做得还是太差。

09年的时候，我开始带项目，项目太紧太忙，20来号人一条心，基本被项目计划推着走，谈不上管理。

10年的时候，开始带部门，工作任务繁重，我每天忙于项目的沟通协调，问题越来越多，越来越忙，然而我察觉到人员技能水平提升缓慢，业务不熟悉，开发技能不熟，有点跟不上，于是我就把部门项目管理和协调工作交给了产品经理（我们这个产品经理很强，据说现在在蚂蚁金服P8），就开始着手做人员培训，开发技能、开发规范基本我来讲，业务知识部分我来讲，也邀请业务部门的同事利用晚上的时间过来给大家讲。当时整个部门的人员都非常团结，加班都不知道累的，培训也是晚上搞。有一天，部门一个女同事跑过来跟我说，她搞出了一个bug，很多商户的结算信息被清掉了，很明显能感觉到她很慌张，我没有责怪，把问题升级，当即和这位同事，同时带了部门3-4个壮丁一起奔到机房处理，利用接口日志来恢复数据，几个人进行分工，进行筛选、合并、加工、复核，搞了一整晚，搞得双眼晕花，没有一个人喊累，大家同心协力在天亮前把问题解决掉了。感觉真好。可是我也有犯蠢的时候，部门里有一个男同事在追女同事，我居然很蠢的跟他们沟通说部门里不要谈恋爱，搞得人家羞愧难当，差点立即辞职。对这个事情，我一直深怀歉意，在这里再说次对不起。

再后来，由于去参加了一次qcon的峰会，发现外面的技术真是强大，被震撼到了，于是确定了先技术后管理的职业规划。回来后立即提出不干部门经理了，我要转去做架构，然后我就开始了长达7年的架构生涯。

我的管理观念的形成

做着架构的活，我开始关注整个团队中的优秀开发苗子，发现一个就立即逮过来做平台研发。这些年来，人员流动较多，但是平台研发的核心还是一直始终保持着。

虽然做着架构的工作，每次管理例会大家反馈的问题，我基本都会认真听，设身处地的思考这些问题的解决办法，也发现管理问题真是五花八门，各种难题层出不穷，期间也买了一些管理书籍，开始慢慢建立自己的一些管理观。

几年前，我在网上看到一篇文章，讲诉的一个企业的领导带领大家共同奋斗，共同成长的心历路程，当时很感动，觉得以后自己做管理也需要有老大的风范。

管理者的职责

我也观察到有一些管理者每天非常忙碌，但部门管理一团糟，根本没有意识到一个技术管理者有三大基本责任：

  • 实施好项目
  • 培养好人才
  • 控制好质量

这三个工作缺一不可，否则就是一个不称职不合格的技术管理。

培养人才

由于一直从事着技术工作和人员挖掘工作，我更加重视人员的培养，建立xx学苑知识库，问答区，致力打造学习氛围。可惜后来由于各种原因没有再专门运营下去了。

优秀的领导者应该像扫冰面的选手一样，竭尽全力地帮助下属扫除前进道路上的障碍。

杰克·韦尔奇用一段话来定义一个领导者的主要职责。

Before you are a leader, success is all about growing yourself.
When you are a leader, success is all about growing others.
在你成为领导者之前，成功的全部就是自我成长。
当你成为领导者之后，成功的全部就变成帮忙他人成长

当员工沮丧时，给员工鼓励；
当员工懒惰时，给员工鞭策；
当员工安逸时，给员工压力；
当员工优秀时，给员工赞美；

给员工明确的目标，协助员工制定职业的规划。

遇见更好的自己

发表于 2018-05-24 | 分类于 管理 |

生命的意义

《活出生命的意义》这是一本对人类有深远影响意义的书，说的是人在极限环境下的苦难，以及从苦难里如何借着意义地寻找，将自己超拔出来，去重新爱人。如果你没看过，可能根本想象、体会不到纳粹集中营那恐怖的死亡弥漫的气息。

作者弗兰克尔在那里遭受了常人无法想象的艰辛和摧残。他的父母、妻子、哥哥，全都死于毒气室中，只有他和妹妹幸存下来。支持他活下来的不是坚信有一天可以从集中营出来的希望，而是他找到了生命的意义，他倡导了“意义疗法”，提出人是由生理、心理和精神三方面的需求满足的交互作用统合而成的整体，生理需求的满足使人存在，心理需求的满足使人快乐，精神需求的满足使人有价值感。

尼采说：知道为什么而活的人，便能生存。

每个人要都自己去寻找生命的意义。寻找生命意义的三个途径：工作（做有意义的事）、爱（关爱他人）以及拥有克服困难的勇气。

自律

知道很多道理，却过不好这一生。很多道理大家都懂，可是真正能做到并坚持的是少之又少。在知乎看到一个挺有意思的网络新词：积极废人。

积极废人指那些喜欢给自己立flag，但是永远做不到的人，这类人心态积极向上，行动却宛如废物，他们往往会在间歇性享乐后恐慌，时常为自己的懒惰自责。

我们给自己定了健身的目标，健身馆的钱也交了，去了几次就坚持不下去了。
年初制定了今年要读多少书的计划，书也买了，放在桌上，到现在塑料膜还没拆。
“积极废人”就是不自律的结果。从去年开始，我给自己定个目标，要努力成为自律的人，要从知道变成做到，我不想做“积极废人”。

  • 2017年：戒烟且成功
  • 2018年：每天1万步
  • 2018年：每天都要读书，至少一个小时，读完一本书要写读后感

最近发现自己有一个坏习惯，经常看一会书就想拿起手机看看，这严重影响到了读书的质量。所以我打算克服掉这个坏习惯：读书的时候把手机丢在客厅。上班的时候，手机也尽量不碰。从现在开始。

战胜两个坏毛病：拖延与抱怨

高效绩效团队的三个秘密读后感

发表于 2018-05-17 | 分类于 管理 |

这本书写的内容不太成体系，后面一些内容有点拼凑。明道公司勇于OKR实践，并把实践的经验分享出来，也毫不避讳实际工作的一些对话，也可以看到一些诚意。虽然算不上什么很好的书，但读一读还是可以补充自己一些认知的不足，也是有收获的。如果你是名管理者，正在物色合适的企业管理软件，可以看看这本书，了解明道的管理理念，或许你会很愿意去尝试一下。

GTD和企业任务管理

职场中只有6%的人能够建立和保持GTD习惯。如果我们要依赖员工的自律来构筑一个任务型组织，成功就太侥幸了

我自己在几年前也实践过GTD，当时做了部门经理，杂事很多，看完《小强升职记》后，发奋了执行GTD一段时间，大概实施了半年，还是没有坚持下去。人性的弱点之一就是惰性。

企业任务并不能简单地分派为个人任务，然后依靠人的自觉性了事。我们不能把人想象为完人，不能幻想依靠个人的自律来完成繁重的任务检查，评估和总结。自动自发的行为当然可贵，但是它难以持续和重复。所以，在讲述任务型组织之前，我们首先明确的是企业任务系统是一个闭环的协同过程，有任务的发起人，单一的负责人（可能就是发起人本人），参与执行的成员，明确的到期日期和完成标准，这些是企业任务的要素。除了这些静态要素，一个完整的任务闭环还可能包括期中的进度检查，任务结果的检查，瑕疵的归因分析，任务成效的评定。简单说，任务除了发起，更加需要检查和评价。没有检查的任务和个人GTD的结果不会有太大差别。

关于留人

你真正的价值，将在下一份工作中体现。大多数成功人士都是换了几次工作之后才获得现在的职位。换一家公司，往往是提升职级最迅速有效的方法。
每个人都渴望成功，都希望出人头地，你不能阻止你的员工也这样想。越是优秀的员工，你越难以留住他，因为他的选择可以有很多。既然留不住，索性就不要去想那件事情了，就把成就一个员工作为你工作的成功吧!
唯一能够留住员工的条件，就是公司的成长速度高于员工个人的成长速度。

是的，要留住人，还是需要有公共的愿景，并且公司成长的。静姐离开了，我难过，难过的是搭档多年，我无法给他创建更大的成长空间。然而有些人是注定要去追逐远方的。不要道德绑架。

建设性的指出“问题”

每个公司都会有问题，它有时候会演变成一场危机，有时候会成为迈向卓越的起点，不同的处理方式会让我们得到不同的结果。
按照“PSI”原则，指出问题，让事情向好的方向转化。
P：problem，指出你看到的问题
S：solution，提出你的解决方案
I：what can i do，对此我能做什么？

不要埋怨，也不要视而不见，正面问题，利用PSI原则推进问题的解决。

社会化日报

我刚参加工作的时候，我的部门经理要求每天都要给写一份工作和学习总结，我坚持了半年，每天都把工作中学到的知识总结成文章，把原理和思路画出来，让leader进行指导和确认。半年内我学完公司所有的技术文档，然后很快被提升为组长。在我离职的时候，我才知道我当初写的一些总结在经理中传播，深受好评。一直以来，我都认为写工作日报和学习总结是非常有用的管理方式，只是这种管理方式多数情况下，让员工不爽，所以也没有去推行。

直到最近看到这本书里介绍了社会化日报的形式，我突然觉得，这种形式的日报是相对容易推广和执行的。以后有机会可以尝试一下。

How Google Works读后感

发表于 2018-05-17 | 分类于 管理 |

Google把企业塑造成一个像学校的环境，里面汇集世界各地的创意精英，拥有一流、舒适、自由的公办环境，有豪华的餐厅、活动场地，确实是打工的天堂，被网上肆意吹捧。国内前几年也涌出一些创业企业，声称拥有google的文化，在创业初期租的市中心的高档办公场地，配备好几千的电脑椅，但没多久由于业务没气色，资金出现问题很快就灭亡了。创业初期应该以业务发展为中心，在业务有起色，有资金流之后，再考虑更舒适的环境，有多大的屁股就下多大的蛋。

《How Google Works 》这本书我一读就爱不释手，坐地铁的时候也拿着看。这是一本非常好的书，五星满星，强烈推荐给大家。
不得不说Google的成功是不可复制的，但是其文化和管理理念我非常的喜欢，是可以借鉴的。

文化

Google每年都会发布一份《创始人公开信》，重新审视公司取得的成就和变化。”着眼于长远”、”为用户服务”、”不作恶”、”让世界更美好”，这些都是让人热血澎湃的伟大使命。

在一次会议上，大家讨论对广告体制做出一项改变可能带来的好处。虽然有可能为公司带来丰厚的利润，但一位工程负责人拍桌反对”这是在作恶，这事我们不能做”，最终这个广告体制的提案被否决了。”不作恶”深深的渗透到google的企业文化中的方方面面。这是很多企业无法做到的。

如果你不能时常传达你的目标，不能通过奖励巩固你的目标，那么，你的愿景还不如打印愿景的纸有价值。
– 杰克.韦尔奇

Redis访问慢的问题

发表于 2017-10-01 | 分类于 技术 |

​ 有一套三主三从的Master-Slave Replication 的Redis集群，突然出现部分访问时间很长，甚至超过五秒。查看redis的服务端日志，发现大量的Asynchronous AOF fsync is taking too long (disk is busy?). Writing the AOF buffer without waiting for fsync to complete, this may slow down Redis.

  1. You ask Redis to fsync every second the AOF, to flush the kernel buffers to disk.
  2. Redis will try, and usually disk will be able to receive the generated I/O without too much problems.
  3. Then Redis has to rewrite the log because it is too big, so Redis creates another child to rewrite the log, however the log rewriting will use additional I/O.
  4. At this point if the disk is slow, the additional I/O used to rewrite the log saturated it to the point that Redis will try to fsync data every second but the background thread is not able to accomplish the sync in a second. Redis will log a warning, and eventually if after two seconds the thread is still waiting to perform the fsync, it starts blocking, in your interest: you want data to be transmitted on disk!

考虑到该redis集群存放的只是数据库提取出来的部分缓存数据，且有主从复制，故直接取消aof。设置appendonly = false，问题解决。
